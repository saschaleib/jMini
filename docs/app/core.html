$app (core)

Core objects and functionalities for the jMini $app framework.

This module is required by the other app modules.

Functions

The core module provides no functions that would typically be of interest for implementators. It’s main purpose is to call the so-called ”pre-init“ of the other loaded $app modules.

Pre-init

The internal pre-init function $app._init() (note the underscore!) gets called when the document’s DOMContentLoaded event is raised (i.e. when the document is initially loaded. This then proceeds and checks all sub-onjects of $app if they contain a function called _init, and call each of these.

To profit of this, each module that needs a pre-init shoud simply implement a function of this name. If it is present, it will be called in this process.

Please note that the order in which the modules will be called is not defined. Implementers should not rely on any other module to be initialized earlier or later.

Global init

After the pre-init phase, the function $app.init() (without an underscore!) will be called, if it exists.

Implementing this function is up to the developer of the specific app, and it should be added as follows:

$app.init = function() { … }

Note: any init functions of specific modules are not automatically called. If modules need initialisations that can not be done in the pre-init phase (for example, because they need specific parameters), these have to be called by the user-script, e.g. from the init() function.

For this to work, the $app object needs to be created at the stage when this code is called. This fails, for example, if the jMini script file is loaded with the defer attribute, but the function definition is inlined into the HTML file.

If you want to use deferred loading (which is strongly recommended!) you can avoid such an error by “baking” your scripts into the jMini file (see Download page). For smaller scripts, or during the development phase, it is probably easier to remove the defer attribute.
